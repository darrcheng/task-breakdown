Two bugs in triggerComplete, both on lines 86-108 of TaskListItem.tsx. Fix both in this single change.

**BUG 1 FIX -- Force-complete regardless of current status (lines 88-89):**

Remove the getNextStatus guard from triggerComplete entirely. The function's purpose is "complete this task" -- it should go straight to done regardless of whether task.status is 'todo', 'in-progress', or even 'done'.

Change:
```typescript
const nextStatus = getNextStatus(task.status);
if (nextStatus === 'done') {
```

To: Remove both lines. The body of the if-block becomes the direct body of triggerComplete (after the `if (!task.id || departing) return;` guard). No conditional needed -- triggerComplete always means "mark done with celebration."

Also update handleStatusClick (lines 138-147): Since triggerComplete no longer checks getNextStatus internally, handleStatusClick should call triggerComplete() for ALL status transitions that target done. The current code already does `if (nextStatus === 'done') { triggerComplete(); }` which is correct -- but now triggerComplete will ALSO work when called directly by the swipe Done button for todo-status tasks.

Also remove `task.status` from the triggerComplete useCallback dependency array since triggerComplete no longer reads task.status. The deps become `[task.id, departing]`.

**BUG 2 FIX -- Move DB write before settling phase (lines 93-105):**

The current timeline is: ring (1500ms) -> settling (removes opacity-0, 400ms) -> setDepartingPhase(null) + DB write.
The settling phase removes opacity-0, causing the task to fade BACK to visible over 300ms. Then 400ms later the DB write removes it from the DOM. This is the flash-back.

Change the inner setTimeout chain so the DB write fires FIRST, then settling handles the visual cleanup for the show-completed=true case:

```typescript
departureTimeout.current = setTimeout(async () => {
  departureTimeout.current = null;
  // DB write FIRST: when showCompleted=false, Dexie liveQuery unmounts
  // the component immediately, so settling never visually matters.
  // When showCompleted=true, the component stays and settling smoothly
  // removes the ring glow.
  setDepartingPhase(null);
  await db.tasks.update(task.id!, {
    status: 'done',
    updatedAt: new Date(),
  });
}, 1500);
```

This removes the settling phase entirely. The rationale: settling was added to prevent abrupt ring disappear, but the ring has already faded to invisible during the fade phase (opacity-0 + 1500ms transition on the ring). By the time 1500ms elapses, both the ring and the element are at opacity 0. There is nothing to "settle" -- the ring is already invisible. So settling is unnecessary and harmful.

Remove the settlingTimeout ref usage from triggerComplete (no longer needed). Keep the cleanup in the unmount useEffect (lines 73-82) in case of future use, but it will be a no-op since settlingTimeout.current will always be null.

Also remove `'settling'` from the departingPhase useState type union: change `useState<'ring' | 'fade' | 'settling' | null>(null)` to `useState<'ring' | 'fade' | null>(null)`.

Remove the settling CSS class line (line 162): `departingPhase === 'settling' && 'transition-all duration-300'` -- delete this line from the clsx() call since the settling phase no longer exists.

5. Remove `'settling'` from departingPhase type union
6. Remove settling CSS class line from clsx()
