# Phase 02.1: Gemini Model Selector — Research

**Researched:** 2026-02-23
**Domain:** Gemini API model selection, React settings UI, localStorage persistence
**Confidence:** HIGH (core implementation), MEDIUM (Gemma streaming/system instruction caveats)

## Summary

This is an entirely contained UI + provider enhancement. The existing codebase already has all the infrastructure needed: `GeminiProvider` takes an `apiKey` and passes a hardcoded `model` string to `generateContentStream`. The only moving parts are: (1) storing the user's model choice in localStorage alongside the provider preference, (2) passing it through to `GeminiProvider` at construction time, (3) surfacing a model picker in `AIProviderSettings` after Gemini is selected.

The four target models are all available via the same `@google/genai` API key — no extra auth or SDK. Flash Lite and Gemini 3 Flash are full Gemini models (support `systemInstruction`). The two Gemma models (`gemma-3-27b-it`, `gemma-3-12b-it`) are open models hosted via the Gemini API; they likely support `generateContentStream` but **system instruction support is unconfirmed** — the safest path is to inline the system prompt as the first user message for Gemma models, or to try `systemInstruction` and fall back.

No new npm packages are needed. The change touches five files: `types/index.ts`, `key-storage.ts` (or a new `model-storage.ts` helper), `providers/gemini.ts`, `hooks/useAIProvider.ts`, and `AIProviderSettings.tsx`.

**Primary recommendation:** Store the Gemini model choice in localStorage under `taskbreaker-gemini-model`. Pass it as a constructor arg to `GeminiProvider`. Render a sub-section of radio/toggle buttons in `AIProviderSettings` when Gemini is the selected provider.

### Anti-Patterns to Avoid

- **Encrypting the model ID:** Unlike the API key, the model ID is not sensitive. Store it plainly in localStorage — no IndexedDB/AES-GCM needed.
- **Re-testing connection on model change:** Model switching is instant; do not prompt the user to re-enter the API key just because the model changed. The existing key is still valid for all models.
- **Treating model change as a provider change:** `configureProvider` should not be called when switching models. Only the `setGeminiModel` setter is needed.
- **Hardcoding the model in the GeminiProvider constructor call in multiple places:** All `createProvider` calls already go through `provider-factory.ts`. Only that factory needs updating.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Model persistence | Custom encrypted store | Plain `localStorage.setItem` | Model IDs are not sensitive |
| Model enum | Runtime string validation | TypeScript union type `GeminiModelId` | Compiler catches invalid strings at dev time |
| Model UI | Custom dropdown/select | Existing button-toggle pattern from SettingsModal | Consistent with established UI — no new component needed |

## Common Pitfalls

### Pitfall 1: Gemma system instruction support uncertain

**What goes wrong:** `GeminiProvider.generateSubtasks` passes `config: { systemInstruction: systemPrompt }`. For Gemma models via the Gemini API, official docs do not confirm `systemInstruction` works the same way as Gemini models. If it silently ignores the system instruction, output format may be inconsistent (the provider relies on JSON-lines format driven by the system prompt).

**Why it happens:** Gemma is an open model hosted on the Gemini API infrastructure. Its chat template differs from Gemini's.

**How to avoid:** Two options:
1. (Simpler) Try `systemInstruction` on Gemma models — if the API accepts it without error, it likely passes through. This is the default approach; the app already handles malformed output gracefully in `parseSubtaskLines`.
2. (Defensive) In `GeminiProvider`, detect if the model starts with `gemma-` and prepend the system instruction to the first user message instead of using `config.systemInstruction`.

**Warning signs:** Gemma model produces raw prose instead of JSON-lines during manual test.

### Pitfall 2: Stale model in getProvider()

**What goes wrong:** `getProvider()` in `useAIProvider` is memoized with `useCallback`. If `state.geminiModel` is not in the dependency array, the provider will be created with the old model even after the user changes it.

**Why it happens:** The existing `getProvider` already has this shape; adding `geminiModel` to state is safe as long as the `useCallback` dep array is updated.

**How to avoid:** Ensure `useCallback([state.provider, state.geminiModel])` — both must be deps.

### Pitfall 3: Model not restored after page refresh

**What goes wrong:** `useAIProvider` initializes provider from localStorage on mount but forgets to also load `geminiModel`.

**Why it happens:** The `useEffect` on mount reads `PROVIDER_STORAGE_KEY` but does not yet know about `GEMINI_MODEL_KEY`.

**How to avoid:** In the same `useEffect`, also call `loadGeminiModel()` and set it in state alongside the provider.

### Pitfall 4: testConnection uses wrong model

**What goes wrong:** `GeminiProvider.testConnection()` today hardcodes `model: 'gemini-2.5-flash'`. After refactor, if the user selects Gemma 3 27B and clicks "Test & Save", the test passes (using the old model) but actual generation fails with the new model.

**Why it happens:** `testConnection` is independent of `generateSubtasks`.

**How to avoid:** Use `this.model` in `testConnection` as well — both methods should use the same instance field.

### Pitfall 5: Model selector visible before Gemini is configured

**What goes wrong:** If model selector appears before the user has saved a Gemini API key, they can change the model but it has no effect until they configure. This is confusing.

**How to avoid:** Only show the model sub-section when `currentProvider === 'gemini' && hasKey`. This mirrors how the "API key saved" indicator works today.

## Code Examples

### Full GeminiProvider after change

```typescript
// Source: existing src/ai/providers/gemini.ts pattern + modification
import { GoogleGenAI } from '@google/genai';
import type { AIProvider, StreamCallbacks, SubtaskSuggestion } from './types';
import { buildSubtaskPrompt } from '../prompts';
import type { GeminiModelId } from '../../types';

const GEMINI_DEFAULT_MODEL: GeminiModelId = 'gemini-2.5-flash';

export class GeminiProvider implements AIProvider {
  name = 'gemini';
  private ai: GoogleGenAI;
  private model: string;

  constructor(apiKey: string, model: string = GEMINI_DEFAULT_MODEL) {
    this.ai = new GoogleGenAI({ apiKey });
    this.model = model;
  }

  async generateSubtasks(
    taskTitle: string,
    taskDescription: string,
    parentContext: string,
    callbacks: StreamCallbacks,
  ): Promise<void> {
    const prompt = buildSubtaskPrompt(taskTitle, taskDescription, parentContext);
    const systemPrompt = 'You are a task breakdown assistant helping users with ADHD break overwhelming tasks into small, actionable steps. Output each subtask as a separate JSON line with "title" and "description" fields. Do not output anything else.';

    try {
      const response = await this.ai.models.generateContentStream({
        model: this.model,
        contents: [{ role: 'user', parts: [{ text: prompt }] }],
        config: {
          systemInstruction: systemPrompt,
        },
      });
      // ... rest unchanged
    } catch (error) {
      // ... rest unchanged (CORS hint, etc.)
    }
  }

  async testConnection(): Promise<boolean> {
    try {
      await this.ai.models.generateContent({
        model: this.model,  // use instance model, not hardcoded
        contents: 'Hi',
      });
      return true;
    } catch {
      return false;
    }
  }
}
```

### localStorage helpers (plain, not encrypted)

```typescript
// Inline in useAIProvider.ts or extracted to src/ai/gemini-model-storage.ts
const GEMINI_MODEL_KEY = 'taskbreaker-gemini-model';

export function saveGeminiModel(model: GeminiModelId): void {
  localStorage.setItem(GEMINI_MODEL_KEY, model);
}

export function loadGeminiModel(): GeminiModelId {
  const saved = localStorage.getItem(GEMINI_MODEL_KEY) as GeminiModelId | null;
  // Validate against known values, fall back to default
  const valid: GeminiModelId[] = [
    'gemini-2.5-flash',
    'gemini-2.5-flash-lite',
    'gemini-3-flash-preview',
    'gemma-3-12b-it',
    'gemma-3-27b-it',
  ];
  return saved && valid.includes(saved) ? saved : 'gemini-2.5-flash';
}
```

## Open Questions

1. **Does `systemInstruction` work with Gemma models via the Gemini API?**
   - What we know: Official `gemma_on_gemini_api` docs show only `generateContent`, not streaming or system instructions
   - What's unclear: Whether `config.systemInstruction` silently applies or is ignored for Gemma
   - Recommendation: Implement with `systemInstruction` for all models first. During testing, if Gemma output is malformed prose (not JSON-lines), add a `this.model.startsWith('gemma-')` branch that prepends system prompt inline.

2. **Is `gemma-3-12b-it` available to all Gemini API tiers?**
   - What we know: `gemma-3-27b-it` is in official docs; `gemma-3-12b-it` appears in API marketplaces and the changelog listing
   - What's unclear: Rate limits or tier restrictions on Gemma 12B via AI Studio free tier
   - Recommendation: Include it; users with free keys will get an API error if unavailable — the existing error handling already surfaces this.

3. **Does the model selector affect the ProviderSetupModal?**
   - What we know: `ProviderSetupModal` calls `configureProvider(selectedProvider, apiKey)` — it does not need to select a model (user can change model in Settings afterward)
   - What's unclear: Nothing — the setup flow only handles provider+key; model selection is post-setup
   - Recommendation: Do not add model selection to `ProviderSetupModal`. Keep the setup flow as: choose provider → enter API key → test & save. Model selector lives only in `AIProviderSettings` (Settings modal).
