Three sub-fixes:

**A. Two-frame departure animation in SubtaskRow (SubtaskList.tsx):**
The CSS transition race happens because `opacity-0` and `transition-all` are applied in the same paint frame, so the browser never sees a transition from opacity-1 to opacity-0.

Fix: Split departure into two phases using state. When `setDeparting(true)` is called:
1. Frame 1: Apply ring/glow classes but NOT opacity-0 yet. Use a new state variable `departingPhase` (`'ring' | 'fade' | null`). Set `departingPhase = 'ring'`.
2. Frame 2: After a `requestAnimationFrame` inside a `useEffect` watching `departingPhase`, set `departingPhase = 'fade'` which adds `opacity-0`.

Update the `clsx` in SubtaskRow's outer div:
- When `departingPhase === 'ring'`: apply `ring-2 ring-emerald-400 ring-offset-1 transition-all duration-[1500ms]` (no opacity-0 yet)
- When `departingPhase === 'fade'`: apply `ring-2 ring-emerald-400 ring-offset-1 opacity-0 transition-all duration-[1500ms]`
- The `departing` boolean can be derived as `departingPhase !== null`.

Also in the departure timeout callback, add `setDeparting(false)` (set departingPhase to null) AFTER the DB write. This ensures that when `showCompleted=true`, the task reappears after the animation instead of staying invisible.

**B. Same two-frame fix in TaskListItem.tsx:**
Apply identical two-frame pattern: `departingPhase` state replaces simple `departing` boolean. Frame 1 applies ring classes, Frame 2 (via requestAnimationFrame in useEffect) adds `opacity-0`. The existing `departing` variable should be derived as `departingPhase !== null` so the re-click cancel logic still works. After DB write in the timeout callback, set departingPhase back to null (setDeparting(false) equivalent).

**C. Filter subtasks from flat list queries in hooks.ts:**
In both `useTasksByDate` and `useTasksByDateRange`, add `!t.parentId` to the filter predicates in BOTH branches (showCompleted and not). This prevents subtasks from appearing as orphan rows in list view.

Change the filter predicates from:
```
t.status !== 'done' && !t.isSomeday && (!energyFilter || t.energyLevel === energyFilter)
```
to:
```
t.status !== 'done' && !t.isSomeday && !t.parentId && (!energyFilter || t.energyLevel === energyFilter)
```

Apply `!t.parentId` to all 4 filter predicates (2 per hook, showCompleted and not).

Two fixes:

**A. Fix stale isConfigured race in useBreakdown.ts:**
The root cause: `isConfigured` is derived from `useAIProvider` state which initializes as `false` until a `useEffect` fires. Clicking "Break it down" before the async init completes hits the `!isConfigured` branch and shows the setup modal.

Fix: In `startBreakdown`, replace the `isConfigured` guard with a direct provider check. Instead of:
```typescript
if (!isConfigured) {
  pendingTaskRef.current = task;
  setState({ status: 'configuring' });
  return;
}
```

Change to:
```typescript
const provider = await getProvider();
if (!provider) {
  pendingTaskRef.current = task;
  setState({ status: 'configuring' });
  return;
}
```

Then use the already-fetched `provider` for the subsequent `generateSubtasks` call instead of calling `getProvider()` again. Remove `isConfigured` from the `useCallback` deps array (replace with just `[getProvider]`). The `isConfigured` variable is still destructured from `useAIProvider` but is no longer used in `startBreakdown`.

**B. Add Enter keyboard shortcut for list view inline create in App.tsx:**
Add a `case 'Enter'` to the global keyboard handler in App.tsx. It should only fire when `viewMode === 'list'`.

Implementation approach: Dispatch a custom DOM event that DayGroup listens for. This avoids tight coupling.

In App.tsx keyboard handler, add after the `case 'n'` block:
```typescript
case 'Enter':
  e.preventDefault();
  if (viewMode === 'list') {
    // Dispatch custom event for today's DayGroup to open inline create
    window.dispatchEvent(new CustomEvent('taskbreaker:inline-create', {
      detail: { date: formatDateKey(new Date()) }
    }));
  }
  break;
```

In DayGroup.tsx, add a `useEffect` that listens for this custom event:
```typescript
useEffect(() => {
  const handleInlineCreate = (e: Event) => {
    const detail = (e as CustomEvent).detail;
    if (detail.date === date) {
      setIsCreating(true);
    }
  };
  window.addEventListener('taskbreaker:inline-create', handleInlineCreate);
  return () => window.removeEventListener('taskbreaker:inline-create', handleInlineCreate);
}, [date]);
```

Also add `src/components/list/DayGroup.tsx` to the modified files for the event listener.
