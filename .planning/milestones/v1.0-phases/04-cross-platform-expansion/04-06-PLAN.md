<task type="auto">
  <name>Task 1: Add stopPropagation to SwipeableTaskRow to prevent day-navigation conflict</name>
  <files>src/components/mobile/SwipeableTaskRow.tsx</files>
  <action>
    The swipe conflict: DaySwipeView uses react-swipeable with delta:50 for day navigation. SwipeableTaskRow also uses react-swipeable with delta:20 for action reveal. When a user swipes on a task row, BOTH handlers fire — the task row reveals actions AND the day advances. Fix this by stopping the underlying touch event from propagating to the parent DaySwipeView handler.

    In SwipeableTaskRow's useSwipeable config, add the `onSwiping` event handler to call `eventData.event.stopPropagation()` at the START of the swiping callback (before the existing direction logic). This prevents the touch event from bubbling up to DaySwipeView's swipe handler.

    Specifically, in the `onSwiping` callback (line ~27), add as the FIRST line:
    ```
    eventData.event.stopPropagation();
    ```

    This ensures that when a user starts swiping on a task row, the DaySwipeView's swipe handler does not also receive the event. Day navigation still works when swiping on empty space or the day header because those areas are outside SwipeableTaskRow.

    Also add `touchEventOptions: { passive: false }` to the useSwipeable config to ensure stopPropagation works reliably on touch events.
  </action>

    <automated>npx tsc --noEmit --pretty 2>&1 | head -20</automated>
  </verify>
  <done>SwipeableTaskRow calls stopPropagation on swiping events, preventing touch event bubbling to parent DaySwipeView swipe handler. TypeScript compiles without errors.</done>

  <action>
    Wire up the missing SwipeableTaskRow wrapper in DaySwipeView.tsx.

    1. Add imports at top of file:
       ```
       import { SwipeableTaskRow } from './SwipeableTaskRow';
       import { db } from '../../db/database';
       ```

    2. In the task list mapping (the `tasks.map((task) => ...)` block around line 68), wrap each TaskListItem in SwipeableTaskRow:
       ```tsx
       {tasks.map((task) => (
         <SwipeableTaskRow
           key={task.id}
           onComplete={async () => {
             if (task.id) {
               await db.tasks.update(task.id, {
                 status: 'done',
                 updatedAt: new Date(),
               });
             }
           }}
           onDelete={async () => {
             if (task.id) {
               await db.tasks.delete(task.id);
             }
           }}
           isCompleted={task.status === 'done'}
         >
           <TaskListItem
             task={task}
             categoryMap={categoryMap}
             onClick={() => onTaskClick(task)}
           />
         </SwipeableTaskRow>
       ))}
       ```

    Note: The `key` prop moves from TaskListItem to SwipeableTaskRow (the outermost element in the map). TaskListItem no longer needs a key since it is a single child.

    The onComplete handler directly sets status to 'done' (matching the pattern in TaskModal.tsx and other delete/complete handlers in the codebase — direct db.tasks.update). The onDelete handler uses db.tasks.delete (matching TaskModal.tsx line 160, SomedayView.tsx line 32).

    Do NOT add hapticFeedback here — the TaskListItem's own handleStatusClick already has haptic feedback for the status cycle flow. The swipe-to-complete is a quick action that bypasses the departure animation.
