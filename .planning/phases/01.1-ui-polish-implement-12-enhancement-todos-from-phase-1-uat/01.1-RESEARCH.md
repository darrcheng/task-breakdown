Phase 1.1 is a pure UI polish sprint — no new dependencies required. Every todo is a targeted modification to an existing component. The full stack (React 19, Tailwind CSS 4, Dexie 4, dnd-kit, lucide-react, clsx, date-fns) is already installed and working.

The 12 todos divide into four natural groups: (1) visual polish to existing UI elements (today indicator, sticky header), (2) interaction pattern improvements (clickable cell, clickable status boxes, blank-space task creation, add button size), (3) new micro-components (keyboard shortcuts, settings modal, popover positioning, combobox category picker, inline category field, status-cycle safety), and (4) a data guard removal (default category deletability). The key architectural constraint is that the codebase avoids external UI component libraries (no Radix, no Headless UI, no shadcn). All UI is hand-crafted in Tailwind + React. The research confirms this is the right call to maintain for Phase 1.1: every todo is achievable in plain React + Tailwind without additional libraries.

**Primary recommendation:** Implement all 12 todos as surgical, file-specific edits. Group related todos into plans by component cluster to minimize merge conflicts.

### Pattern 2: Keyboard Shortcuts (Todo #2)

**What:** Global `keydown` listener in App.tsx using `useEffect`. Map: J=next period, K=prev period, T=today, M=month view, W=week view (or calendar view toggle), C=create task for today.
**When to use:** Applied once at App level, not in individual nav components.

```tsx
// App.tsx
useEffect(() => {
  const handleKeyDown = (e: KeyboardEvent) => {
    // Don't fire when user is typing in an input/textarea/select
    const tag = (e.target as HTMLElement).tagName;
    if (['INPUT', 'TEXTAREA', 'SELECT'].includes(tag)) return;
    if (e.metaKey || e.ctrlKey || e.altKey) return;

    switch (e.key) {
      case 'j': // Next period
        setCurrentMonth(prev =>
          calendarView === 'week' ? addWeeks(prev, 1) : addMonths(prev, 1)
        );
        break;
      case 'k': // Previous period
        setCurrentMonth(prev =>
          calendarView === 'week' ? subWeeks(prev, 1) : subMonths(prev, 1)
        );
        break;
      case 't': // Today
        setCurrentMonth(new Date());
        break;
      case 'm': // Month view
        setCalendarView('month');
        setViewMode('calendar');
        break;
      case 'w': // Week view
        setCalendarView('week');
        setViewMode('calendar');
        break;
      case 'c': // Create task
        setModalState({ isOpen: true, date: formatDateKey(new Date()) });
        break;
    }
  };

  window.addEventListener('keydown', handleKeyDown);
  return () => window.removeEventListener('keydown', handleKeyDown);
}, [calendarView]); // calendarView needed for j/k to know period size
```

**Critical pitfall:** The `calendarView` dependency means the effect re-runs on view change. This is correct but must be handled — use `useCallback` or include all state refs properly.

### Pattern 3: Settings Modal (Todo #3)

**What:** A new modal component `SettingsModal.tsx` rendered in App.tsx. Stores settings in `localStorage` (no DB needed — it's device preferences). Initial settings: start-of-week (Sun/Mon), keyboard shortcut display toggle.
**Structure:**

```tsx
// src/components/ui/SettingsModal.tsx
interface Settings {
  weekStartsOn: 0 | 1; // 0=Sunday, 1=Monday
  showKeyboardShortcuts: boolean; // whether to show shortcuts in UI hints
}
```

Settings are stored in localStorage as JSON under key `'taskbreaker-settings'`. A custom hook `useSettings()` reads/writes this. The `weekStartsOn` setting must be threaded through to `getCalendarDays()` and `getWeekDays()` in `utils/dates.ts` which currently hardcode Sunday start.

**date-fns support for weekStartsOn:** date-fns `startOfWeek` and `eachDayOfInterval` accept `{ weekStartsOn: 0 | 1 | 2 | 3 | 4 | 5 | 6 }` options. The existing `getCalendarDays` in `utils/dates.ts` must accept this parameter and pass it to `startOfWeek`.

### Pattern 4: Popover Near Clicked Date (Todo #4)

**What:** Instead of a centered `<dialog>` (which centers itself via UA stylesheet + CSS), position the modal near the clicked DayCell.
**Implementation approach:** Native `<dialog>` does not support positional anchoring natively without the CSS Anchor Positioning API (not yet broadly supported). The cleanest approach for this project:

1. Pass a `clickPosition: { x: number; y: number }` from `DayCell.onDayClick` up to App.tsx
2. In `TaskModal.tsx`, apply inline `style` to position the dialog content (the inner `div`) absolutely near the position, with bounds-clamping to stay on screen
3. Keep the `<dialog>` element itself for backdrop/modal semantics, but override the default centering

```tsx
// DayCell.tsx — pass click coordinates
const handleClick = (e: React.MouseEvent) => {
  e.stopPropagation();
  onDayClick(dateStr, { x: e.clientX, y: e.clientY });
};
```

```tsx
// App.tsx ModalState
interface ModalState {
  isOpen: boolean;
  date: string;
  task?: Task;
  clickPosition?: { x: number; y: number };
}
```

```tsx
// TaskModal.tsx — position near click
const dialogStyle = clickPosition ? {
  position: 'fixed' as const,
  left: Math.min(clickPosition.x, window.innerWidth - 400),
  top: Math.min(clickPosition.y, window.innerHeight - 500),
  margin: 0,
} : {};
```

**Alternative:** Use a plain `div` with `position: fixed` and a semi-transparent backdrop overlay instead of `<dialog>`. This is simpler for positioning. Given the project uses `<dialog>` for TaskModal and a `div` overlay for CategoryManager, using the same `div`-based approach (as CategoryManager uses) for a positioned popover is the cleanest.

### Pattern 5: Full Cell Clickable (Todo #5)

**What:** The entire DayCell area (not just the date number div) triggers `onDayClick`. Currently the `onClick` is on a child `div` wrapping everything including the task cards. DraggableTasks must NOT bubble clicks to the day handler.

**Current structure:**
```tsx
<DroppableDay>
  <div onClick={() => onDayClick(dateStr)}>   {/* ← this div covers tasks too */}
    <div className="date number">...</div>
    <div className="tasks">...</div>
  </div>
</DroppableDay>
```

**Fix:** Move the onClick to the DroppableDay level (or the outer wrapper), and have TaskCard/DraggableTask call `e.stopPropagation()` on click. This way clicking blank space = day click, clicking a task = task click.

```tsx
// DayCell.tsx
<DroppableDay
  dateStr={dateStr}
  className={...}
  onClick={() => onDayClick(dateStr)}    // outer click = create
>
  <div className="date number" onClick={...} />
  {tasks.map(task => (
    <DraggableTask key={task.id} task={task}>
      <TaskCard
        task={task}
        categoryMap={categoryMap}
        onClick={(t) => { /* stopPropagation happens in TaskCard */ onTaskClick(t); }}
      />
    </DraggableTask>
  ))}
</DroppableDay>
```

TaskCard's onClick must call `e.stopPropagation()` to prevent bubbling to the day.

### Pattern 7: Searchable Category Combobox (Todo #7)

**What:** Replace the `<select>` category dropdown in TaskForm with a combobox: text input that filters categories by name, shows matching list, "Create [query]" option if no exact match. Recent categories shown first.

**Implementation approach (no external library):**

```tsx
// Inline combobox in TaskForm.tsx or extracted as CategoryCombobox.tsx
const [search, setSearch] = useState('');
const [isOpen, setIsOpen] = useState(false);

const filtered = useMemo(() => {
  const q = search.toLowerCase();
  return (categories ?? []).filter(c =>
    c.name.toLowerCase().includes(q)
  );
}, [categories, search]);

const showCreate = search.trim() && !filtered.some(
  c => c.name.toLowerCase() === search.trim().toLowerCase()
);
```

The "recent first" ordering requires tracking recently-used category IDs. Simplest approach: store an array of recently-used category IDs in localStorage under `'taskbreaker-recent-categories'`. On category select, prepend its ID to the list (max 3 stored). Sort the displayed list: recent IDs first, then alphabetical for the rest.

**Accessibility note:** Use `role="combobox"`, `aria-expanded`, `aria-autocomplete="list"` on the input. The dropdown list gets `role="listbox"` and each option gets `role="option"` with `aria-selected`. Keyboard: ArrowDown/Up to navigate, Enter to select, Escape to close.

**Create from search:** When user clicks "Create [query]", call `db.categories.add({ name: search.trim(), icon: 'folder', isDefault: false })` and set the returned ID as `categoryId`.

### Pattern 11: Status Cycling Safety (Todo #11)

**What:** In `TaskListItem.tsx`, when a task is `done`, clicking the status circle should NOT immediately cycle back to `todo`. Instead, show a confirmation state first (e.g., the button glows or shows an undo hint for 2 seconds before actually cycling).

**Implementation options:**
1. **Two-click confirm for done→todo:** Same pattern as the delete confirmation. Click once on `done` = enters "confirm reopen" state for 2s. Click again = cycles. Timeout resets.
2. **Single click with undo toast:** Cycle immediately but show a brief "Undo" button for 3 seconds.

Option 1 is simpler and matches the existing delete-confirm pattern in the codebase:

```tsx
// TaskListItem.tsx
const [confirmReopen, setConfirmReopen] = useState(false);

const handleStatusClick = async (e: React.MouseEvent) => {
  e.stopPropagation();
  if (task.id) {
    if (task.status === 'done' && !confirmReopen) {
      // First click on done: enter confirm state
      setConfirmReopen(true);
      setTimeout(() => setConfirmReopen(false), 2000);
      return;
    }
    setConfirmReopen(false);
    await db.tasks.update(task.id, {
      status: getNextStatus(task.status),
      updatedAt: new Date(),
    });
  }
};
```

The button gets a visual indicator when `confirmReopen` is true (e.g., pulsing ring or different color with tooltip "Click again to reopen").

**What:** Remove the `!cat.isDefault` guard in `CategoryManager.tsx` that hides the delete button for default categories. Allow deletion of all categories including defaults.

**Implementation:**

**Data concern:** Tasks reference `categoryId`. If a default category is deleted, existing tasks will have a dangling `categoryId`. Handle this by either:

**Also:** Remove the "Default" badge display or change it to something non-restrictive. Keep the badge if desired — just remove the delete restriction.

### Anti-Patterns to Avoid

- **Installing Radix UI or Headless UI for the combobox:** Overkill, adds significant bundle weight. A 30-line controlled filter list in React is sufficient.
- **Using CSS `position: fixed` with `top: 50%; left: 50%; transform: translate(-50%, -50%)` for the popover:** This is the current dialog centering. The fix must override this with actual coordinates.
- **Keyboard shortcut handler that listens to `keypress` instead of `keydown`:** `keypress` is deprecated. Use `keydown`.
- **Global keyboard handler in a child component:** Always attach global keyboard shortcuts in App.tsx, not in MonthNavigation or DayCell.
- **Making sticky header work by adding `overflow: hidden` to a parent:** This breaks sticky positioning. The sticky container must not have `overflow: hidden/auto/scroll` on any ancestor.
- **Forgetting to call `e.stopPropagation()` in TaskCard after making DayCell clickable:** This will cause every task card click to also trigger a "create new task" modal.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Date arithmetic for week start | Custom week-start math | date-fns `startOfWeek({ weekStartsOn })` | date-fns already handles all edge cases |
| Keyboard shortcut library | Custom event map | Plain `useEffect` + `addEventListener` | 6 shortcuts don't need a library |
| Combobox component library | Radix Combobox | Controlled input + filtered list | ~30 lines, no additional dep |
| Tooltip/popover positioning | Popper.js / Floating UI | `clientX/Y` + CSS `min()` clamping | Simple fixed positioning sufficient for this use case |
| Settings persistence | Custom state management | `localStorage` with JSON | No sync needed, device-local preference |

**Key insight:** Every todo in this phase is a targeted UI change. The complexity is in the interaction design, not the data structures. Avoid over-engineering by reaching for external libraries.

## Common Pitfalls

### Pitfall 1: Sticky Header Breaks Sticky Day Headers in List View

**What goes wrong:** Adding `sticky top-0` to the app header makes list view's DayGroup headers (`sticky top-0 z-10`) overlap BEHIND the app header — they were calibrated for `top: 0` from the viewport top, but now the viewport top is occupied by the header.

**Why it happens:** Sticky elements use their nearest scrolling ancestor's top edge. The DayGroup headers are sticky within ListView's scroll container. The app header is outside that scroll container. So they don't actually interfere — UNLESS the scroll container starts at the top of the viewport and the header sits above it in the same stacking context.

**How to avoid:** After making the header sticky, test list view and verify day headers remain visible below the app header. If they stack behind, either:
1. Add `top-[57px]` to DayGroup sticky headers to offset by header height, OR
2. Ensure the ListView scroll container has `padding-top` matching the header height.

**Warning signs:** Day headers disappear when scrolling in list view after sticky header is applied.

### Pitfall 2: DayCell Click Creates Task When Dragging

**What goes wrong:** Making the full DayCell clickable (todo #5) conflicts with dnd-kit's drag behavior. When a user drags a task card and releases over a day, the click event may fire, triggering a "create task" modal.

**Why it happens:** Drag-and-drop libraries suppress pointer events but don't always suppress click events on release. The `onClick` on the cell fires after `mouseup`.

**How to avoid:** dnd-kit's `DragEndEvent` fires before any click. Use a ref or state flag `isDragging` that is set on `DragStart` and cleared after a brief delay after `DragEnd`. In the day click handler, check this flag.

**Warning signs:** After dragging a task, a create-task modal unexpectedly opens.

### Pitfall 3: Combobox State Reset on Category List Update

**What goes wrong:** `useCategories()` returns a reactive Dexie query. When the user creates a new category from the combobox, the `categories` array updates, which can reset the combobox search state if not handled carefully.

**Why it happens:** React re-renders the component when `useLiveQuery` updates, and if the combobox `useState` for search is reset (e.g., via key or effect), the user loses their typed query.

**How to avoid:** Keep the combobox search state independent of the `categories` live query. Only close the combobox and set `categoryId` after the create operation completes. Don't reset `search` on `categories` change.

### Pitfall 4: Keyboard Shortcuts Fire in Modals

**What goes wrong:** The global keyboard handler in App.tsx fires even when the TaskModal or SettingsModal is open. Pressing 'c' while the modal is open could open a second create-task flow. Pressing 'j'/'k' navigates the calendar while the user is filling in form fields.

**Why it happens:** The check `if (['INPUT', 'TEXTAREA', 'SELECT'].includes(tag)) return` only guards against target being an input. But the modal can have focus on the dialog itself, not an input.

**How to avoid:** Add a check: `if (modalState.isOpen || isCategoryManagerOpen || isSettingsOpen) return;` at the top of the keyboard handler. Or check if a `<dialog>` element is open: `if (document.querySelector('dialog[open]')) return`.

### Pitfall 5: Popover Overflows Viewport

**What goes wrong:** Positioning the task modal near the clicked cell using raw `clientX, clientY` causes the modal to overflow off-screen for cells near the right edge or bottom of the viewport.

**Why it happens:** The modal is ~400px wide and ~500px tall. Cells near the right edge of the calendar have `clientX` values of e.g. 1400px on a 1440px viewport, causing `left: 1400` = 360px overflow.

**How to avoid:**
```tsx
const MODAL_WIDTH = 400;
const MODAL_HEIGHT = 500;
const left = Math.min(clickPosition.x + 8, window.innerWidth - MODAL_WIDTH - 16);
const top = Math.min(clickPosition.y + 8, window.innerHeight - MODAL_HEIGHT - 16);
```

### Pitfall 6: Settings weekStartsOn Not Propagated to Calendar

**What goes wrong:** The settings modal saves `weekStartsOn` to localStorage, but the CalendarGrid/WeekView/date utilities still hardcode Sunday start.

**Why it happens:** `getCalendarDays()` in `utils/dates.ts` calls `startOfWeek(date)` without `weekStartsOn` option, defaulting to locale (Sunday).

**How to avoid:** Update `getCalendarDays(month, weekStartsOn = 0)` to accept the param and pass it through. Thread the settings value from App down to CalendarGrid via props. The CalendarGrid `DAY_NAMES` header array must also rotate to start from the correct day.

### Settings Hook

```tsx
// src/hooks/useSettings.ts
const SETTINGS_KEY = 'taskbreaker-settings';

interface Settings {
  weekStartsOn: 0 | 1;
  showKeyboardShortcuts: boolean;
}

const DEFAULT_SETTINGS: Settings = {
  weekStartsOn: 0,
  showKeyboardShortcuts: true,
};

export function useSettings() {
  const [settings, setSettings] = useState<Settings>(() => {
    try {
      const stored = localStorage.getItem(SETTINGS_KEY);
      return stored ? { ...DEFAULT_SETTINGS, ...JSON.parse(stored) } : DEFAULT_SETTINGS;
    } catch {
      return DEFAULT_SETTINGS;
    }
  });

  const updateSettings = (updates: Partial<Settings>) => {
    setSettings(prev => {
      const next = { ...prev, ...updates };
      localStorage.setItem(SETTINGS_KEY, JSON.stringify(next));
      return next;
    });
  };

  return { settings, updateSettings };
}
```

```tsx
  onChange: (id: number) => void;
}) {
  const categories = useCategories();
  const [search, setSearch] = useState('');
  const [isOpen, setIsOpen] = useState(false);

  const selectedCat = categories?.find(c => c.id === value);

  const filtered = useMemo(() => {
    if (!categories) return [];
    const q = search.toLowerCase().trim();
    if (!q) return categories;
    return categories.filter(c => c.name.toLowerCase().includes(q));
  }, [categories, search]);

  const showCreate = search.trim() &&
    !filtered.some(c => c.name.toLowerCase() === search.trim().toLowerCase());

  const handleSelect = (id: number) => {
    onChange(id);
    setSearch('');
    setIsOpen(false);
  };

  const handleCreate = async () => {
    const id = await db.categories.add({
      name: search.trim(),
      icon: 'folder',
      isDefault: false,
    });
    onChange(id as number);
    setSearch('');
    setIsOpen(false);
  };

  return (
    <div className="relative">
      <input
        type="text"
        value={isOpen ? search : (selectedCat?.name ?? '')}
        onChange={e => { setSearch(e.target.value); setIsOpen(true); }}
        onFocus={() => setIsOpen(true)}
        onBlur={() => setTimeout(() => setIsOpen(false), 150)}
        placeholder="Search categories..."
        className="w-full px-3 py-2 border border-slate-300 rounded-md text-sm ..."
        role="combobox"
        aria-expanded={isOpen}
      />
      {isOpen && (
        <ul
          className="absolute z-50 w-full mt-1 bg-white border border-slate-200 rounded-md shadow-lg max-h-48 overflow-y-auto"
          role="listbox"
        >
          {filtered.map(cat => (
            <li
              key={cat.id}
              onMouseDown={() => handleSelect(cat.id!)}
              className="flex items-center gap-2 px-3 py-2 hover:bg-slate-50 cursor-pointer text-sm"
              role="option"
              aria-selected={cat.id === value}
            >
              <CategoryIcon icon={cat.icon} className="w-4 h-4" />
              {cat.name}
            </li>
          ))}
          {showCreate && (
            <li
              onMouseDown={handleCreate}
              className="flex items-center gap-2 px-3 py-2 hover:bg-blue-50 cursor-pointer text-sm text-blue-600 border-t border-slate-100"
            >
              <Plus className="w-4 h-4" />
              Create "{search.trim()}"
            </li>
          )}
        </ul>
      )}
    </div>
  );
}
```

**Note on `onBlur` with `setTimeout`:** The 150ms delay on blur prevents the dropdown from closing before `onMouseDown` on a list item fires. This is a standard pattern for click-inside-blur scenarios.

   - What we know: UAT noted "Start-of-week preference, keyboard shortcut config"
   - What's unclear: Should shortcut config allow rebinding keys, or just toggle shortcuts on/off?
   - Recommendation: Toggle on/off only for v1. Rebinding is complex and not requested.

2. **Category combobox: "recent first" implementation**
   - What we know: UAT said "recent first"
   - What's unclear: Is "recent" based on recently-selected or recently-created?
   - Recommendation: Recently-selected (store in localStorage). Simplest meaningful interpretation.

3. **Popover positioning: which direction to open?**
   - What we know: Open "near clicked date"
   - What's unclear: Should it prefer opening below/right, or above/left for cells near edges?
   - Recommendation: Prefer below-right (Google Calendar style), with clamp to keep on-screen.

4. **Status cycling safety: 2-click confirm or undo toast?**
   - What we know: UAT said "protection against accidental done→todo cycling"
   - What's unclear: Which UX pattern is preferred?
   - Recommendation: 2-click confirm (matches existing delete pattern in codebase, no timeout UX needed).

5. **DayGroup blank-space click: which area exactly?**
   - What we know: "click blank space to create"
   - What's unclear: Does this mean the empty area below tasks, or anywhere in the day group that isn't a task?
   - Recommendation: The task list container div (below existing tasks), checking `e.target === e.currentTarget`.
