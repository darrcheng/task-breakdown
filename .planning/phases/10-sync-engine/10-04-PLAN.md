          ```typescript
          interface AuthState {
            user: User | null;
            loading: boolean;
            syncing: boolean;  // true during migration
          }
          ```

       b. Initialize with `syncing: false` in useState default.

       c. Import `startSync`, `stopSync`, `migrateLocalData`, `isMigrating` from `../firebase/sync`.

       d. In the `onAuthStateChanged` callback:
          - When `user` becomes non-null (sign-in):
            1. Set `syncing: true`
            2. Call `await migrateLocalData(user.uid)` -- uploads existing local data
            3. Call `startSync(user.uid)` -- starts onSnapshot listeners (inbound sync begins, pulling any cloud-only data)
            4. Set `syncing: false`
            5. Set `loading: false`
          - When `user` becomes null (sign-out):
            1. `stopSync()` is already called in SettingsModal before reaching here
            2. Just set state: `{ user: null, loading: false, syncing: false }`

       e. The async operations in onAuthStateChanged callback: wrap in an async IIFE or use `.then()` since the callback itself can't be async. Pattern:
          ```typescript
          onAuthStateChanged(auth, (user) => {
            if (user) {
              setState({ user, loading: true, syncing: true });
              migrateLocalData(user.uid)
                .then(() => {
                  startSync(user.uid);
                  setState({ user, loading: false, syncing: false });
                })
                .catch((err) => {
                  console.error('Sync setup failed:', err);
                  setState({ user, loading: false, syncing: false });
                  // App still works -- just without sync
                });
            } else {
              setState({ user: null, loading: false, syncing: false });
            }
          });
          ```

       f. Expose `syncing` in context value so components can show "Setting up sync..." spinner.

<task type="auto">
  <name>Task 2: Handle category deduplication after sync</name>
  <files>src/db/database.ts</files>
  <action>
    1. The problem: `db.on('populate')` seeds 5 default categories when Dexie DB is created (first use or after `db.delete()` on sign-out). When a user signs in on a second device, onSnapshot will deliver Firestore categories, but the Dexie populate event has already seeded defaults, causing duplicates.

    2. The cleanest fix per CONTEXT.md and RESEARCH.md: Let populate seed defaults as normal. When inbound sync arrives with categories from Firestore, the onSnapshot handler in sync.ts will put each category with its original numeric ID. Since the Firestore category IDs match the original Dexie IDs (from migration), `db.categories.put()` with the same ID will OVERWRITE the existing category -- no duplicates.

    3. The edge case: if the user signed in on Device A (which had categories 1-5), those got uploaded to Firestore with IDs "1"-"5". Device B signs in, populate seeds local categories 1-5, then inbound sync arrives with IDs 1-5 from Firestore -- they overwrite perfectly. NO code change needed for this case.

    4. The other edge case: User creates a custom category (ID 6) on Device A. Device B signs in, populate seeds 1-5, sync brings 1-6 from Firestore. Category 6 is added via `db.categories.put({...data, id: 6})` -- no conflict since ID 6 doesn't exist locally yet. Still no issue.

    5. Verify by reading the inbound onSnapshot handler in sync.ts:
       - Confirm it uses `db.categories.put()` (upsert by ID), not `db.categories.add()` (insert)
       - If it uses `put`, no dedup code needed -- IDs match and overwrite naturally
       - If for some reason the IDs don't match (shouldn't happen with String(id) -> Number(docId) mapping), add a dedup step

    6. Add a comment in `src/db/database.ts` near the `on('populate')` handler explaining why dedup isn't needed:
       ```
       // Default categories seeded here will be overwritten by Firestore sync
       // when onSnapshot delivers categories with matching IDs. No dedup needed
       // because sync uses db.categories.put() which upserts by primary key.
       ```

    7. IMPORTANT per CONTEXT.md: "Only seed default categories if Firestore has no categories (first device ever)". The current populate behavior seeds defaults on every fresh DB (including after sign-out wipe). This is fine because:
       - If Firestore has categories, they'll overwrite these defaults via sync
       - If Firestore has no categories (first device), these defaults are correct and will be migrated to Firestore

    8. Run all tests and build: `npx vitest run --reporter=verbose && npm run build`
